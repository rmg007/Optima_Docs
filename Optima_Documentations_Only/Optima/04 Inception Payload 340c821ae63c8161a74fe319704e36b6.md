# 04 Inception Payload

This document contains the exact content Optima generates in host projects. Copy these strings verbatim into the generator modules.

## File: `.claude/rules/optima-feedback.md`

Generated by `src/generator/feedback-rules.ts`. Written on first `optima_get_context` call and regenerated when the feedback rules need updating.

```markdown
---
paths: "**/*"
---

## Optima Memory Rules

These rules ensure Optima learns from every session to improve future ones.

### After Fixing Errors

When you successfully resolve a compiler error, runtime exception, or failing test:
- You MUST call the `optima_memorize` tool before concluding your response.
- Include:
  - `type`: `"error_fix"`
  - `error`: The original error message (copy the key part, not the full stack trace)
  - `resolution`: A concise summary of what fixed it
  - `files`: Array of file paths that were modified
  - `directory`: The primary directory involved

### After Making Changes

After modifying code, always verify your work before reporting completion:
- Run the project's test command (if known from `optima_get_context`)
- Run the project's lint command (if known)
- If tests or lint fail, fix the issues before calling `optima_memorize`
- Only log the fix after verification passes

### After Architectural Decisions

When you make a significant design choice (new pattern, structural decision, technology choice):
- Call `optima_memorize` with:
  - `type`: `"architectural_rule"`
  - `rule`: The decision in one sentence
  - `rationale`: Why this approach was chosen
  - `directory`: The directory scope (or omit for project-wide)

### At Task Start

When beginning any task (bug fix, feature, refactor, test, review):
- Call `optima_get_context` with the relevant directory path
- Use the returned context (gotchas, rules, entities) to inform your approach
- Pay special attention to gotchas — they contain solutions to previously encountered errors

### After Establishing Patterns

When you establish a reusable code pattern or convention:
- Call `optima_memorize` with:
  - `type`: `"pattern"`
  - `pattern`: Description of the pattern
  - `example`: A brief code example
  - `directory`: Where this pattern applies
```

## File: `CLAUDE.md` (Section Markers)

Generated by `src/generator/claude-md.ts`. Uses section markers to avoid overwriting human-authored content.

**Instruction budget constraint:** Claude Code's [CLAUDE.md](http://CLAUDE.md) has a practical ceiling of ~150 distinct instructions before adherence degrades. Optima's generated sections must stay well under this ceiling, assuming the developer has their own hand-written instructions too. Target: Optima sections should contribute no more than **30-40 instructions** total (leaving 100+ slots for the developer's own content). This means:

- `project_overview`: ~5 instructions (name, stack, commands)
- `architecture_rules`: Cap at 10 rules (most relevant by recency and scope)
- `known_gotchas`: Cap at 10 gotchas (sorted by `hit_count` desc)
- `patterns`: Cap at 5 patterns (most established ones)
- **Total: ~30 instructions** — safe within the ceiling

**The necessity test (applied during generation):** Every instruction must pass: "Would Claude Code make a concrete mistake without this line?" Apply these filters:

- Gotchas with `hit_count` of 0 that haven't triggered in 30+ days: **exclude** from [CLAUDE.md](http://CLAUDE.md) (keep in database for future matching).
- Rules that duplicate what the project's linter/formatter already enforces (detected via config file presence — `eslint.config.*`, `.prettierrc`, `biome.json`, `ruff.toml`, `pyproject.toml [tool.ruff]`, `.editorconfig`): **exclude entirely**.
- Architecture rules that are obvious from the codebase (e.g., "uses TypeScript" when `tsconfig.json` exists, "uses React" when `react` is in dependencies): **exclude**.
- Patterns with only 1 occurrence and no `hit_count`: **exclude** until confirmed by reuse.

**Domain-aware framing:** The project analyzer extracts the project's purpose from `README.md` (first paragraph), `package.json` description field, or `pyproject.toml` description. If detected (e.g., "payment processing API", "game engine", "internal dashboard"), the `project_overview` section includes a `Purpose` line. This sets Claude Code's risk appetite — a fintech app gets defensive coding emphasis, a hackathon prototype gets speed emphasis. Example: `- **Purpose**: B2B invoicing platform — data accuracy and auditability are top priorities.`

**If [CLAUDE.md](http://CLAUDE.md) does not exist:** Create it with all Optima sections.

**If [CLAUDE.md](http://CLAUDE.md) already exists:** Append Optima sections at the end, below any existing content. Never modify content above the first Optima marker.

**Section marker format:**

```markdown
<!-- OPTIMA:START section_id -->
(generated content here)
<!-- OPTIMA:END section_id -->
```

## Regeneration Logic

1. Read existing [CLAUDE.md](http://CLAUDE.md) content (if file exists).
2. For each section ID, find the `<!-- OPTIMA:START section_id -->` and `<!-- OPTIMA:END section_id -->` markers.
3. Replace content between markers with freshly generated content.
4. If a section’s markers don’t exist yet, append the section at the end of the file.
5. Content outside of Optima markers is never modified.
6. Compute SHA-256 hash of the final content. Compare with `generation_log`. Skip write if unchanged.

Created on first run. Contains:


No other files should be created in `.optima/`. All state lives in the database.

## Phase 2: Hook-Based Feedback (Future)

In Phase 2, Optima can generate `.claude/settings.json` hook configurations that fire deterministically alongside the advisory rules. Unlike rules (which the LLM may not follow), hooks execute automatically at lifecycle points.

**Example hook configuration** (generated into `.claude/settings.json`):

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "echo '{\"file\": \"$TOOL_INPUT_FILE_PATH\", \"timestamp\": '$(date +%s)'}' >> .optima/edit-log.jsonl",
            "timeout": 5
          }
        ]
      }
    ]
  }
}
```

This creates a local edit log that Optima reads on the next `optima_get_context` call — passive file-change tracking without a file watcher. Combined with the rules-based MCP tool calls, this gives Optima a dual-layer feedback system: deterministic event logging (hooks) + semantic knowledge capture (rules → `optima_memorize`).

**Phase 2 agent memory:** Generated `.claude/agents/` files should include `memory: project` in frontmatter so agent knowledge persists in `.claude/agent-memory/<agent-name>/` and is shareable via git.

> **DO NOT implement hooks in MVP.** The rules-only inception pattern is sufficient and simpler.
