# 01 Product Spec MVP

## What Optima Is

Optima is an MCP server for Claude Code. It runs locally via Stdio transport, analyzes the developer’s project, remembers solutions to errors, and maintains intelligence files (`CLAUDE.md`, `.claude/rules/`) that make Claude Code smarter with every session.

## How It Works

1. Developer configures Optima once in `.mcp.json` (or `~/.claude.json` globally).
2. Claude Code connects to Optima automatically on session start.
3. Claude Code calls `optima_get_context` — Optima lazily indexes the project and returns structured context.
4. Claude Code works with full project awareness.
5. `.claude/rules/optima-feedback.md` (generated by Optima) instructs Claude Code to call `optima_memorize` after fixing errors or making decisions.
6. Optima stores the knowledge in its SQLite database.
7. Optima regenerates `CLAUDE.md` with accumulated knowledge.
8. Next session is smarter than the last.

> **Why this works:** Claude Code re-reads [CLAUDE.md](http://CLAUDE.md) at the start of every new session and after every `/compact` or `/clear` command. This means Optima's regenerated [CLAUDE.md](http://CLAUDE.md) is always picked up automatically — there's no cache invalidation problem. The developer's session quality compounds without any manual intervention.
> 

## The Inception Pattern

MCP servers cannot passively observe Claude Code conversations. They only activate when called. Optima solves this by generating `.claude/rules/optima-feedback.md` — a rules file that instructs Claude Code when and how to call Optima’s tools. Optima writes the instructions that teach its consumer how to feed it data. The intelligence layer bootstraps its own feedback loop.

> **Design note:** Rules are advisory (the LLM decides whether to follow them). Claude Code also supports deterministic **hooks** in `.claude/settings.json` that fire automatically at lifecycle points (`PostToolUse`, `TaskCompleted`, etc.). For MVP, rules are sufficient and simpler. Phase 2 adds hook-based automation as a dual-layer enhancement.
> 

## Lazy Indexing Lifecycle

Indexing is NOT exposed as a tool Claude Code calls explicitly. It happens internally inside `optima_get_context`:

1. Claude Code calls `optima_get_context({path: "src/auth/"})`.
2. Optima checks `fs.stat` mtime for every file under `src/auth/` against the `mtime_ms` column in `file_index`.
3. Files with newer mtimes are re-indexed (content hashed, entities re-extracted via Tree-sitter).
4. Unchanged files are skipped.
5. Context is assembled from the fresh index and returned.

The only exception is `optima_reindex` — a manual escape hatch for full project re-indexing after cloning, branch switching, or major restructuring.

## MVP Tool Surface

Three tools. No more.

| Tool | Purpose | When Called |
| --- | --- | --- |
| `optima_get_context` | Returns project context with lazy re-indexing | Start of any task |
| `optima_memorize` | Stores error fixes, rules, patterns, preferences | After completing a task |
| `optima_reindex` | Forces full project re-index | Rarely — after clone/branch switch |

## Generated Files

Optima generates and maintains these files in the host project:

| File | Purpose |
| --- | --- |
| `CLAUDE.md` | Project memory — tech stack, architecture rules, top gotchas (section markers preserve human content) |
| `.claude/rules/optima-feedback.md` | The inception loop — instructs Claude Code to call Optima tools |
| `.gitignore` addition | Adds `.optima/` to gitignore on first run |

## Claude Code Integration Safety

**Optima reads but does NOT modify:** `.claude/settings.json`, `package.json`, `tsconfig.json`, `pyproject.toml` (used for project analysis and tech stack detection).

**Optima NEVER reads or indexes:** `~/.claude/session-env/` (plaintext secrets), `~/.claude/projects/` (session transcripts), `~/.claude/file-history/` (edit snapshots), `CLAUDE.local.md` (developer's personal overrides), `.claude/settings.local.json` (personal permissions), `.env` files, or any Claude Code internal operational state. See `00_START_HERE.md` for the full exclusion list.

**Version control for Optima-generated files:** All files Optima generates are designed to be committed to git (shared with the team):

- `CLAUDE.md` — Committed. Optima’s sections use markers and coexist with human content.
- `.claude/rules/optima-feedback.md` — Committed. Team-shared feedback rules.
- `.optima/` directory — Gitignored (local learning database, not shared).

Optima adds `.optima/` to `.gitignore` on first run. The `.claude/rules/` directory is part of Claude Code's committed project config.

**Settings hierarchy awareness:** Claude Code uses a 5-tier settings priority (managed → CLI → local → shared → global). Optima only writes to `.claude/rules/` (additive, safe) in MVP. Phase 2 hook generation targets `.claude/settings.json` (tier 4, team-shared) with careful JSON merging.

## MVP Included

- MCP server with Stdio transport connects to Claude Code
- `optima_get_context` with lazy re-indexing via mtime checks
- `optima_memorize` with unified input (error_fix, architectural_rule, pattern, preference)
- `optima_reindex` for full project re-index
- Project analyzer: detect tech stack, build/test/lint commands, project domain/purpose (from README/package.json description), and linter/formatter presence (ESLint, Prettier, Biome — suppresses style rules in generated [CLAUDE.md](http://CLAUDE.md))
- File indexer: walk project (respecting .gitignore), track mtimes, compute hashes
- Entity extractor: Tree-sitter parsing for TypeScript (functions, classes, interfaces, types, exports)
- Gotcha Ledger: error → solution storage with normalized dedup
- Rules store: architectural rules and patterns with directory scoping
- [CLAUDE.md](http://CLAUDE.md) generator with section ownership markers
- `.claude/rules/optima-feedback.md` generator (the inception loop)
- `.gitignore` management
- SQLite database with Drizzle schema and versioning
- Single language support: TypeScript
- Dogfood: Optima manages its own [CLAUDE.md](http://CLAUDE.md)
- 80%+ test coverage

## MVP Excluded (Phase 2+)

Do not implement any of the following:

- Additional language support (Python, Dart, Go, etc.)
- `.claude/commands/` generation
- `.claude/skills/` generation
- `.claude/agents/` generation (model-aware sub-agents)
- Token optimization and relevance scoring
- Adaptive prompt templates
- Developer style/verbosity learning
- Historical edit tracking and quality scoring
- Dependency and security auditing
- Test discovery and coverage mapping
- Multi-project support
- Team features

## Success Criteria (Week One)

1. Claude Code calls `optima_get_context` at the start of tasks without being prompted.
2. Claude Code calls `optima_memorize` after fixing errors without being prompted.
3. The Gotcha Ledger contains at least 10 real error → fix mappings.
4. [CLAUDE.md](http://CLAUDE.md) is automatically kept in sync with the project’s actual state.
5. A repeated error is resolved faster because Optima provides the historical fix.
