# 01 Product Spec MVP

## What Optima Is

Optima is an MCP server **focused on Claude Code CLI**. It runs locally via Stdio transport, analyzes the developer‚Äôs project, remembers solutions to errors, and maintains intelligence files (`CLAUDE.md`, `.claude/rules/`) that make Claude Code smarter with every session. Sibling project for VS Code Copilot: [Roadie](https://github.com/rmg007/Roadie).

## Installation (Plug & Play)

### Quick Start

```bash
npm run install:mcp    # One-command installation
# Then restart Claude Desktop / Claude Code sessions
```

The installer handles:
- Cross-platform config discovery (Windows, macOS, Linux)
- Automatic build if not already built
- Smoke-test verification (verifies 5 tools register correctly)
- Atomic config writes with backup creation
- Registration in both Claude Desktop and Claude Code simultaneously

Run `npm run doctor` anytime to verify the installation is healthy.

### Advanced Installation

For more control, use the installer flags:

```bash
npm run install:mcp --desktop      # Claude Desktop only
npm run install:mcp --code         # Claude Code only
npm run install:mcp --log-level DEBUG  # Override log level (default: DEBUG)
npm run uninstall:mcp              # Remove from both clients
```

### Manual Configuration (Fallback)

If the installer fails, manually add Optima to your Claude Code config (`~/.claude.json` or `<project>/.claude/settings.json`):

```json
{
  "mcpServers": {
    "optima": {
      "command": "node",
      "args": ["/absolute/path/to/optima/dist/index.js"],
      "env": {
        "OPTIMA_LOG_LEVEL": "DEBUG"
      }
    }
  }
}
```

Then restart Claude Desktop/Code for the server to be spawned on session start.

## How It Works

### Installation Phase

1. Developer runs `npm run install:mcp` in the Optima repo.
2. Installer automatically:
   - Builds Optima if `dist/index.js` doesn't exist
   - Smoke-tests the server to verify all 5 tools register
   - Discovers Claude Desktop & Code config paths (cross-platform)
   - Writes Optima entry to both configs atomically with backup
3. Developer restarts Claude Desktop / Claude Code sessions.
4. stdio MCP servers are spawned at session start ‚Äî Optima is now available.

### Runtime Phase

5. Claude Code calls `optima_get_context` at the start of a task ‚Äî Optima lazily indexes the project and returns structured context.
6. Claude Code works with full project awareness (entities, gotchas, rules, recent changes).
7. `.claude/rules/optima-feedback.md` (generated by Optima on first run) instructs Claude Code to call `optima_memorize` after fixing errors or making decisions.
8. Developer calls `npm run doctor` anytime to verify the installation is healthy.
9. Optima stores knowledge in its local SQLite database.
10. Optima regenerates `CLAUDE.md` with accumulated knowledge.
11. Next session is smarter than the last.

> **Why this works:** 
> - The installer is **idempotent** and **atomic** ‚Äî safe to re-run, non-destructive backups, no corruption risk
> - The installer **smoke-tests** to catch failures early
> - Claude Code re-reads [CLAUDE.md](http://CLAUDE.md) at the start of every new session and after every `/compact` or `/clear` command. Optima's regenerated [CLAUDE.md](http://CLAUDE.md) is always picked up automatically ‚Äî there's no cache invalidation problem.
> - The developer's session quality compounds without any manual intervention.
> 

## Autonomous Hooks Layer

The bulk of Optima's learning loop runs deterministically through Claude Code lifecycle hooks registered in `~/.claude/settings.json`:

| Event | Matcher | Invokes | Purpose |
| --- | --- | --- | --- |
| `SessionStart` | ‚Äî | `optima-hook prime --project ${CWD}` | Lazy re-index (mtime-delta); refreshes `project_meta`. |
| `PostToolUse` | `Edit|Write|MultiEdit` | `optima-hook observe --tool ${TOOL} --file ${FILE}` | Records an `edit_events` row. |
| `Stop` | ‚Äî | `optima-hook reconcile --project ${CWD}` | Increments `hit_count` on overlapping gotchas, inserts `task_outcomes` on heuristic task completion. |

The hook CLI (`bin/optima-hook.ts`, built to `dist/optima-hook.js`) is a separate entry from the MCP server, opens `.optima/optima.db` directly, and shares `src/db/`, `src/indexer/`, `src/memory/` modules. Contract: always exits 0, writes nothing to stdout, wraps everything in try/catch. Hooks cannot fail Claude Code workflows, and must never corrupt the MCP JSON-RPC stream.

The installer registers all three hooks via `scripts/install.js` (append-only merge ‚Äî existing hook entries are preserved). Use `--no-hooks` to skip.

## The Inception Pattern

MCP servers cannot passively observe Claude Code conversations. They only activate when called. Optima solves this with a two-layer strategy:

1. **Hooks layer (deterministic):** the autonomous capture loop above ‚Äî edits, session transitions, task completion ‚Äî runs without any LLM involvement.
2. **Rules layer (advisory):** `.claude/rules/optima-feedback.md` covers the two moments where the LLM's judgement is required ‚Äî starting work in a new directory (`optima_get_context`) and memorializing a non-trivial error fix (`optima_memorize`).

### Cold Start Bootstrap Sequence

The inception pattern has an apparent circular dependency: Claude Code needs the rules file to know to call Optima, but Optima needs a tool call to generate the rules file. Here is how the bootstrap resolves:

1. **Developer configures `.mcp.json`** ‚Üí Claude Code connects to Optima on next session start.
2. **First session:** Claude Code discovers Optima‚Äôs tools via MCP Tool Search. The tool description for `optima_get_context` ("Get project context, code entities, known errors...") semantically matches most development tasks. Claude Code loads the tool and calls it.
3. **First `optima_get_context` call triggers full cold start:**
   - a. Create `.optima/` directory if it doesn‚Äôt exist.
   - b. Create and open `.optima/optima.db`. Run schema migrations (idempotent ‚Äî checks `schema_version` table).
   - c. Run full project analysis (tech stack, commands, purpose detection).
   - d. Run full file walk + Tree-sitter entity extraction.
   - e. Generate `.claude/rules/optima-feedback.md` (the inception payload). This file instructs Claude Code to use all 5 tools: `optima_get_context`, `optima_memorize`, `optima_reindex`, `optima_dismiss_warning`, and `optima_forget`.
   - f. Generate `CLAUDE.md` with initial Optima sections.
   - g. Append `.optima/` to `.gitignore`.
   - h. Return `GetContextOutput` with full project context.
4. **Same session continues** using the returned context. The feedback rules file exists but Claude Code won‚Äôt re-read `.claude/rules/` mid-session ‚Äî rules are loaded at session start.
5. **Second session onward:** Claude Code reads `.claude/rules/optima-feedback.md` at session start. The rules instruct it to call `optima_get_context` at task start and `optima_memorize` after fixes. The feedback loop is now self-sustaining.

> **Key insight:** The bootstrap doesn‚Äôt require the rules file to exist before the first call. Claude Code‚Äôs Tool Search discovers `optima_get_context` based on its description. The rules file ensures consistent behavior from session 2 onward.


## Lazy Indexing Lifecycle

Indexing is NOT exposed as a tool Claude Code calls explicitly. It happens internally inside `optima_get_context`:

1. Claude Code calls `optima_get_context({path: "src/auth/"})`.
2. Optima checks `fs.stat` mtime for every file under `src/auth/` against the `mtime_ms` column in `file_index`.
3. Files with newer mtimes are re-indexed (content hashed, entities re-extracted via Tree-sitter).
4. Unchanged files are skipped.
5. Context is assembled from the fresh index and returned.

The only exception is `optima_reindex` ‚Äî a manual escape hatch for full project re-indexing after cloning, branch switching, or major restructuring.

### Recent Changes Detection

`GetContextOutput.recent_changes` returns file paths that were re-indexed during the current `optima_get_context` call ‚Äî files whose `mtime_ms` changed since last index, plus files deleted from disk and cleaned from the index. This is the canonical definition (see Doc 03 `optima_get_context` steps 5-7 for the authoritative implementation). Sorted by mtime descending (most recently modified first). Capped at 20 entries. On cold start (no prior index), this list is empty ‚Äî the entire project is new, so "recent changes" is meaningless.

## MVP Tool Surface

Five tools. No more.

| Tool | Purpose | When Called |
| --- | --- | --- |
| `optima_get_context` | Returns project context with lazy re-indexing | Start of any task |
| `optima_memorize` | Stores error fixes, rules, patterns, preferences | After completing a task |
| `optima_reindex` | Forces full project re-index | Rarely ‚Äî after clone/branch switch |
| `optima_dismiss_warning` | Marks a security finding as a false positive | When a security warning is a test fixture or rotated secret |
| `optima_forget` | Deletes a stale or incorrect memory | When a gotcha, rule, or task outcome is outdated |

## Generated Files

Optima generates and maintains these files in the host project:

| File | Purpose |
| --- | --- |
| `CLAUDE.md` | Project memory ‚Äî tech stack, architecture rules, top gotchas (section markers preserve human content) |
| `.claude/rules/optima-feedback.md` | The inception loop ‚Äî instructs Claude Code to call Optima tools |
| `.gitignore` addition | Adds `.optima/` to gitignore on first run |

## Claude Code Integration Safety

**Optima reads but does NOT modify:** `.claude/settings.json`, `package.json`, `tsconfig.json`, `pyproject.toml` (used for project analysis and tech stack detection).

**Optima NEVER reads or indexes:**
- `~/.claude/session-env/` (plaintext secrets injected by Claude Code)
- `~/.claude/projects/` (session transcripts)
- `~/.claude/file-history/` (edit snapshots)
- `CLAUDE.local.md` (developer's personal overrides ‚Äî committed files are indexed, local overrides are not)
- `.claude/settings.local.json` (personal permissions)
- `.env`, `.env.local`, `.env.development`, `.env.production`, `.env.staging`, `.env.development.local`, `.env.production.local` (all `.env.*` variants)
- `node_modules/`, `.git/`, `.optima/`, `dist/`, `build/`, `out/` directories
- `**/secrets/` at any depth
- All binary file formats (images, fonts, media, archives, compiled binaries, databases ‚Äî full list in `02_data_model_and_schema.md`)

**Safe to index (common misconception):** `.env.example` and `.env.template` are NOT excluded ‚Äî they are safe to index as they contain placeholder values, not real secrets.

**Version control for Optima-generated files:** All files Optima generates are designed to be committed to git (shared with the team):

- `CLAUDE.md` ‚Äî Committed. Optima‚Äôs sections use markers and coexist with human content.
- `.claude/rules/optima-feedback.md` ‚Äî Committed. Team-shared feedback rules.
- `.optima/` directory ‚Äî Gitignored (local learning database, not shared).

Optima adds `.optima/` to `.gitignore` on first run. The `.claude/rules/` directory is part of Claude Code's committed project config.

**Settings hierarchy awareness:** Claude Code uses a 5-tier settings priority (managed ‚Üí CLI ‚Üí local ‚Üí shared ‚Üí global). Optima only writes to `.claude/rules/` (additive, safe) in MVP. Phase 2 hook generation targets `.claude/settings.json` (tier 4, team-shared) with careful JSON merging.

## MVP Included

**Core MCP Tools:**
- MCP server with Stdio transport connects to Claude Code
- `optima_get_context` with lazy re-indexing via mtime checks
- `optima_memorize` with unified input (error_fix, architectural_rule, pattern, preference, task_outcome)
- `optima_reindex` for full project re-index
- `optima_dismiss_warning` ‚Äî marks security findings as dismissed (false positive / test fixture / rotated secret)
- `optima_forget` ‚Äî deletes a stale memory (gotcha, rule, or task outcome) with CLAUDE.md regeneration

**Installation & Health Check:**
- `scripts/install.js` ‚Äî One-command plug-and-play installer for Claude Desktop & Code; also registers lifecycle hooks in `~/.claude/settings.json` (skip with `--no-hooks`)
- `scripts/doctor.js` ‚Äî Health check utility (5 checks: build, smoke test, Desktop config, Code config, hooks)
- `npm run install:mcp` ‚Äî Primary installation method (replaces manual config)
- `npm run uninstall:mcp` ‚Äî Clean uninstall from both Claude clients and remove hook entries
- `npm run doctor` ‚Äî Verify installation health

**Autonomous Hooks Layer:**
- `bin/optima-hook.ts` ‚Üí `dist/optima-hook.js` ‚Äî Claude Code lifecycle hook bridge with `prime`, `observe`, `reconcile` subcommands
- Migration `004_edit_events` ‚Äî adds `edit_events(id, file_id, path, tool_name, ts_ms, session_id)` table
- Append-only merge of hook entries in `~/.claude/settings.json` (SessionStart / PostToolUse / Stop)

**Indexing & Analysis:**
- Security scanner: dismissed findings preserved on re-index ‚Äî dismissed state survives full reindex via save/restore
- Structured JSON logging to stderr via `src/utils/logger.ts` (controlled by `OPTIMA_LOG_LEVEL` env var, default: DEBUG)
- Project analyzer: detect tech stack, build/test/lint commands, project domain/purpose (from README/package.json description), and linter/formatter presence (see authoritative list in `02_data_model_and_schema.md` ‚Äî suppresses style rules in generated [CLAUDE.md](http://CLAUDE.md))
- File indexer: walk project (respecting .gitignore), track mtimes, compute hashes
- Entity extractor: Tree-sitter parsing for TypeScript (functions, classes, interfaces, types, exports)

**Knowledge Storage & Retrieval:**
- Gotcha Ledger: error ‚Üí solution storage with normalized dedup
- Rules store: architectural rules and patterns with directory scoping
- [CLAUDE.md](http://CLAUDE.md) generator with section ownership markers
- `.claude/rules/optima-feedback.md` generator (the inception loop)

**Project Configuration:**
- `.gitignore` management
- SQLite database with Drizzle schema and versioning
- Single language support: TypeScript and JavaScript (`.ts`, `.tsx`, `.js`, `.jsx`)
- Dogfood: Optima manages its own [CLAUDE.md](http://CLAUDE.md)
- 80%+ test coverage

## MVP Excluded (Phase 2+)

Do not implement any of the following:

- Additional language support (Python, Dart, Go, etc.)
- `.claude/commands/` generation
- `.claude/skills/` generation
- `.claude/agents/` generation (model-aware sub-agents)
- Token optimization and relevance scoring
- Adaptive prompt templates
- Developer style/verbosity learning
- Historical edit tracking and quality scoring
- Dependency and security auditing
- Test discovery and coverage mapping
- Multi-project support
- Team features

## Success Criteria (Week One)

1. Claude Code calls `optima_get_context` at the start of tasks without being prompted.
2. Claude Code calls `optima_memorize` after fixing errors without being prompted.
3. The Gotcha Ledger contains at least 10 real error ‚Üí fix mappings.
4. [CLAUDE.md](http://CLAUDE.md) is automatically kept in sync with the project‚Äôs actual state.
5. A repeated error is resolved faster because Optima provides the historical fix.
