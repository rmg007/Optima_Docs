```markdown
## Optima Memory Rules

These rules ensure Optima learns from every session to improve future ones.

### After Fixing Errors

When you successfully resolve a compiler error, runtime exception, or failing test:
- You MUST call the `optima_memorize` tool before concluding your response.
- Include:
  - `type`: `"error_fix"`
  - `error`: The original error message (copy the key part, not the full stack trace)
  - `resolution`: A concise summary of what fixed it
  - `files`: Array of file paths that were modified
  - `directory`: The primary directory involved

### After Making Changes

After modifying code, always verify your work before reporting completion:
- Run the project's test command (if known from `optima_get_context`)
- Run the project's lint command (if known)
- If tests or lint fail, fix the issues before calling `optima_memorize`
- Only log the fix after verification passes

### After Architectural Decisions

When you make a significant design choice (new pattern, structural decision, technology choice):
- Call `optima_memorize` with:
  - `type`: `"architectural_rule"`
  - `rule`: The decision in one sentence
  - `rationale`: Why this approach was chosen
  - `directory`: The directory scope (or omit for project-wide)

### At Task Start

When beginning any task (bug fix, feature, refactor, test, review):
- Call `optima_get_context` with the relevant directory path
- Use the returned context (gotchas, rules, entities) to inform your approach
- Pay special attention to gotchas — they contain solutions to previously encountered errors

### After Establishing Patterns

When you establish a reusable code pattern or convention:
- Call `optima_memorize` with:
  - `type`: `"pattern"`
  - `pattern`: Description of the pattern
  - `example`: A brief code example
  - `directory`: Where this pattern applies
```

## File: `CLAUDE.md` (Section Markers)

Generated by `src/generator/claude-md.ts`. Uses section markers to avoid overwriting human-authored content. **Regeneration triggers:** (1) After every `optima_memorize` call (all types — error_fix, architectural_rule, pattern, preference). (2) After every `optima_reindex` call. (3) On cold start (first `optima_get_context`). Content hash check prevents unnecessary file writes — regeneration is cheap.

**Instruction budget constraint:** Optima sections should contribute no more than **30-35 instructions** total:
- `project_overview`: ~5 instructions
- `architecture_rules`: Cap at 10 rules
- `known_gotchas`: Cap at 10 gotchas
- `patterns`: Cap at 5 patterns
- `preferences`: Cap at 5 preferences (if any exist)

**The necessity test (filtering rules for CLAUDE.md inclusion):**
1. **Gotchas:** Include only if `hit_count >= 2` (resolved Q4 — two hits prove value). Additionally, exclude gotchas with `hit_count = 0` AND `created_at` older than 30 days (stale, never triggered).
2. **Rules:** Exclude rules that duplicate linter config (if `linterDetected` is non-null, skip style/formatting rules). Exclude obvious conventions the model knows from general training.
3. **Patterns:** Exclude patterns with only 1 recorded occurrence (not yet established).
4. **Preferences:** Include all (user explicitly stored these; they are always intentional).

**Instruction budget enforcement algorithm:**
1. Query all gotchas/rules/patterns/preferences from database.
2. Apply necessity test filters above.
3. Sort gotchas by `hit_count` DESC, then `updated_at` DESC.
4. Sort rules by `created_at` DESC (most recent first).
5. Sort patterns by occurrence count DESC (if tracked), else `created_at` DESC.
6. Truncate each category to its cap: 10 rules, 10 gotchas, 5 patterns, 5 preferences.
7. If total instructions > 35: reduce lowest-priority items from largest section first.
8. Sections with zero entries after filtering: omit entirely (no empty headers).

**Domain-aware framing:** Extract project purpose from package.json → pyproject.toml → README.md first paragraph.

**Marker format:**

```markdown
<!-- OPTIMA:START section_id -->
(generated content here)
<!-- OPTIMA:END section_id -->
```

## Regeneration Logic

1. Read existing `CLAUDE.md` content.
2. For each section ID, find and replace content between markers.
3. If markers don't exist, append section at end.
4. Content outside markers is never modified.
5. Compute SHA-256 hash and skip write if unchanged.

**Malformed marker handling:**
- **Orphaned START without END:** Replace from START to next marker/EOF with full section.
- **Orphaned END without START:** Delete orphaned END, append full section with both markers.
- **Duplicate markers:** Update only the FIRST pair, delete duplicates.
- **Whitespace variations:** Match with `<!--\s*OPTIMA:START\s+(\w+)\s*-->`.

**Empty sections:** Omit entire section if zero entries.

If none found, `project_purpose` is null.
